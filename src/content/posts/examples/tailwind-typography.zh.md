> 本文来自 [TailwindLabs](https://tailwindcss-typography.vercel.app/)。我将这篇文章放在这里，是为了演示如何使用 AstroPaper 主题来编写博客文章。

默认情况下，Tailwind 会移除段落、标题、列表等元素的所有默认浏览器样式。这在构建应用 UI 时非常有用，因为你不需要花时间去撤销用户代理样式，但当你确实只是想为来自 CMS 富文本编辑器或 Markdown 文件的内容设置样式时，这种做法可能会让人感到意外和反直觉。

实际上，我们收到了很多关于这个问题的抱怨，经常有人问我们这样的问题：

> 为什么 Tailwind 要移除我的 `h1` 元素的默认样式？我该怎么禁用它？你说我还会丢失所有其他基础样式是什么意思？
> 我们听到了你的声音，但我们不认为简单地禁用我们的基础样式是你真正想要的。你肯定不希望每次在仪表板 UI 中使用 `p` 元素时都要手动去除烦人的外边距。而且我怀疑你也不希望你的博客文章使用用户代理样式——你希望它们看起来**很棒**，而不是很糟糕。

`@tailwindcss/typography` 插件是我们为了给你真正想要的东西而做出的尝试，而且不会带来任何像禁用基础样式那样愚蠢的副作用。

它添加了一个新的 `prose` 类，你可以将它应用到任何普通的 HTML 内容块上，将其变成一个美观、格式良好的文档：

```html
<article class="prose">
  <h1>芝士蒜香面包：科学告诉我们什么</h1>
  <p>
    多年来，父母们一直在向孩子们宣扬吃芝士蒜香面包的健康益处，这种食物在我们的文化中获得了如此标志性的地位，以至于孩子们经常在万圣节打扮成温暖的芝士面包。
  </p>
  <p>
    但最近的一项研究表明，这道备受赞誉的开胃菜可能与全国各地出现的一系列狂犬病病例有关。
  </p>
  <!-- ... -->
</article>
```

有关如何使用该插件及其包含的功能的更多信息，请[阅读文档](https://github.com/tailwindcss/typography/blob/master/README.md)。

## 如果标题堆叠在一起呢？

### 我们也应该确保那看起来不错。

有时候标题会直接上下挨在一起。在这种情况下，你通常需要取消第二个标题的上边距，因为标题之间靠得更近通常比段落后跟标题的间距看起来更好。

### 当标题出现在段落之后……

当标题出现在段落之后时，我们需要稍多一点的间距，就像我上面提到的那样。现在让我们看看更复杂的列表是什么样子的。

- **我经常做这样的事情，让列表项带有标题。**

  出于某种原因，我觉得这看起来很酷，但不幸的是，要让样式正确显示相当令人头疼。

  我在这些列表项中经常也有两到三个段落，所以难点在于让段落、列表项标题和不同列表项之间的间距都协调合理。说实话挺难的，你完全可以有力地论证你根本就不应该这样写。

- **既然这是一个列表，我至少需要两个项目。**

  我在上一个列表项中已经解释过我在做什么了，但如果列表只有一个项目，那它就不叫列表了，而我们真的希望这看起来逼真。这就是为什么我添加了这第二个列表项，这样我在写样式的时候就有实际的东西可以参考了。

- **加第三个项目也不是坏主意。**

  我觉得只用两个项目可能也没问题，但三个肯定不会更差，而且既然我似乎可以毫无困难地编造任意的东西来打字，那不如就加上吧。

在这种列表之后，我通常会有一个结束语或段落，因为直接跳到标题看起来有点怪。

## 代码默认应该看起来还行。

我想大多数人如果想给自己的代码块设置样式，都会使用 [highlight.js](https://highlightjs.org/) 或 [Prism](https://prismjs.com/) 之类的工具，但让它们默认看起来还_不错_也没什么坏处，即使没有语法高亮。

以下是撰写时默认的 `tailwind.config.js` 文件的样子：

```js
module.exports = {
  purge: [],
  theme: {
    extend: {},
  },
  variants: {},
  plugins: [],
};
```

希望这对你来说看起来足够好了。

### 嵌套列表呢？

嵌套列表基本上总是很难看，这就是为什么像 Medium 这样的编辑器甚至不让你这么做，但我想既然你们这些家伙中的一些人非要这么干，我们至少得承担起让它能正常工作的责任。

1. **嵌套列表很少是个好主意。**
   - 你可能会觉得自己很"有条理"之类的，但你只是在屏幕上创造了一个难以阅读的丑陋形状。
   - UI 中的嵌套导航也是个坏主意，尽量保持扁平。
   - 在你的源代码中嵌套一大堆文件夹同样没有帮助。
2. **既然我们需要更多项目，这是另一个。**
   - 我不确定我们会不会费心去处理超过两层的嵌套样式。
   - 两层已经太多了，三层绝对是个坏主意。
   - 如果你嵌套四层，你应该进监狱。
3. **两个项目不算真正的列表，三个就好了。**
   - 再说一遍，如果你希望别人真正阅读你的内容，请不要嵌套列表。
   - 没人想看这个。
   - 我们居然还要费心去给这个设置样式，我感到很沮丧。

Markdown 中关于列表最烦人的事情是，除非列表项中有多个段落，否则 `<li>` 元素不会获得子 `<p>` 标签。这意味着我还得操心处理那种烦人的情况。

### 有时候我甚至在标题中使用 `code`

尽管这可能是个坏主意，而且历史上我一直很难让它看起来好看。不过这个_"把代码块用反引号包起来"_的技巧确实效果不错。

我过去做过的另一件事是把 `code` 标签放在链接里面，比如如果我想告诉你关于 [`tailwindcss/docs`](https://github.com/tailwindcss/docs) 仓库。我不太喜欢反引号下面有下划线，但这绝对不值得为了消除它而去折腾那些疯狂的骚操作。

#### 我们还没用过 `h4`

但现在我们用了。请不要在你的内容中使用 `h5` 或 `h6`，Medium 只支持两级标题是有原因的，你们这些野蛮人。我认真考虑过使用 `before` 伪元素来对你大吼大叫，如果你用了 `h5` 或 `h6` 的话。

我们默认完全不给它们设置样式，因为 `h4` 元素已经很小了，和正文一样大。我们还能对 `h5` 做什么，让它比正文还_小_吗？不了，谢谢。
