## 目录

## 创建博客文章

要编写一篇新的博客文章，请在 `src/content/posts/` 目录下创建一个 markdown（或 MDX）文件。

你可以将文章组织到子目录中，以便于内容管理。子目录的名称会成为文章 URL 的一部分。例如，`src/content/posts/2025/example-post.md` 对应的访问地址是 `/posts/2025/example-post`。

如果你只想用子目录来组织文件，而不希望它影响 URL，可以在文件夹名称前加上下划线（`_`）。

```bash
# 示例：文章文件路径及其对应的 URL
src/content/posts/very-first-post.md          -> mysite.com/posts/very-first-post
src/content/posts/2025/example-post.md        -> mysite.com/posts/2025/example-post
src/content/posts/_2026/another-post.md       -> mysite.com/posts/another-post
src/content/posts/docs/_legacy/how-to.md      -> mysite.com/posts/docs/how-to
src/content/posts/Example Dir/Dummy Post.md   -> mysite.com/posts/example-dir/dummy-post
```

> [!TIP]
> 以 `_` 为前缀的文件和目录会被排除在路由之外。你可以利用它们来存放草稿、共享资源或仅限内部使用的内容。

## Frontmatter

Frontmatter 是存储博客文章元数据的主要位置，位于文件顶部的 YAML 格式区域。有关 frontmatter 及其用法的更多信息，请参阅 [Astro 文档](https://docs.astro.build/en/guides/markdown-content/)。

以下是每篇文章的 frontmatter 属性列表：

<ResponsiveTable variant="striped-minimal">

| 属性                 | 描述                                                                                   | 备注                                      |
| -------------------- | -------------------------------------------------------------------------------------- | ----------------------------------------- |
| **_title_**          | 文章标题（h1）                                                                         | 必填<sup>\*</sup>                         |
| **_description_**    | 文章描述，用于文章摘要和文章页面描述                                                   | 必填<sup>\*</sup>                         |
| **_pubDatetime_**    | 发布时间的 ISO 8601 格式                                                               | 必填<sup>\*</sup>                         |
| **_modDatetime_**    | 修改时间的 ISO 8601 格式（仅在文章被修改时添加此属性）                                 | 可选                                      |
| **_author_**         | 文章作者                                                                               | 默认值 = `site.author`                    |
| **_featured_**       | 是否在首页精选区域展示该文章                                                           | 默认值 = false                            |
| **_draft_**          | 将该文章标记为"未发布"                                                                 | 默认值 = false                            |
| **_tags_**           | 文章的相关关键词，以 YAML 数组格式编写                                                 | 默认值 = others                           |
| **_ogImage_**        | 文章的 OG 图片，用于社交媒体分享和 SEO，可以是远程 URL 或相对于当前文件夹的图片路径    | 默认值 = `site.ogImage` 或自动生成的 OG 图 |
| **_canonicalURL_**   | 规范 URL（绝对路径），当文章已在其他来源存在时使用                                     | 默认值 = `Astro.site` + `Astro.url.pathname` |
| **_hideEditPost_**   | 隐藏文章标题下方的编辑按钮，仅对当前文章生效                                           | 默认值 = false                            |
| **_timezone_**       | 以 IANA 格式为当前文章指定时区，仅覆盖此文章的全局 `site.timezone` 配置                | 默认值 = `site.timezone`                  |

</ResponsiveTable>

> [!TIP]
> 你可以在控制台中运行 `new Date().toISOString()` 来获取 ISO 8601 格式的日期时间。

Frontmatter 中只有 `title`、`description` 和 `pubDatetime` 这三个字段是必须指定的。

标题和描述（摘要）对搜索引擎优化（SEO）非常重要，因此 AstroPaper 建议你在所有博客文章中都包含这些内容。

如果你在博客文章中省略了 `tags`（即没有指定任何标签），则该文章将使用默认标签 `others`。你可以在 `src/content.config.ts` 中设置默认标签：

```ts file="src/content.config.ts"
// ...
tags: z.array(z.string()).default(["others"]), // 将 "others" 替换为你想要的默认标签
// ...
```

### Frontmatter 示例

以下是一篇文章的 frontmatter 示例。

```yaml file="src/content/posts/sample-post.md"
---
title: The title of the post
author: your name
pubDatetime: 2022-09-21T05:17:19Z
featured: true
draft: false
tags:
  - some
  - example
  - tags
ogImage: ../../assets/images/example.png # src/assets/images/example.png
# ogImage: "https://example.org/remote-image.png" # 远程 URL
description: This is the example description of the example post.
canonicalURL: https://example.org/my-article-was-already-posted-here
---
```

### VS Code 代码片段（可选）

AstroPaper 内置了工作区代码片段，以加快创建新文章的速度：

- **frontmatter**：插入推荐的 frontmatter 代码块
- **template**：插入基本的文章模板（包含 `## 目录`）

这些代码片段存放在 `.vscode/astro-paper.code-snippets` 中。如果你使用 VS Code（或 Cursor），它们会在你打开工作区时自动可用。

## Callout 提示框

AstroPaper 从 v6.1 开始支持 callout 提示框。它们使用简单的引用块语法，由 `rehype-callouts`（Obsidian 主题）驱动。

以下是最常用的类型：

> [!NOTE]
> 读者应该了解的补充信息。

> [!TIP]
> 有用的建议、快捷方式或最佳实践。

> [!WARNING]
> 可能出现问题或产生意外后果的事项。

> [!DANGER]
> 严重的失败、数据丢失或错误行为风险。

> [!INFO]
> 中性的信息性内容——紧急程度低于注释。

> [!SUCCESS]
> 确认某操作成功或正确。

支持的完整类型列表包括：`NOTE`、`ABSTRACT`、`INFO`、`TODO`、`TIP`、`SUCCESS`、`QUESTION`、`WARNING`、`FAILURE`、`DANGER`、`BUG`、`EXAMPLE`、`QUOTE`——每种都有各自的图标和颜色。许多类型还支持别名（如 `HINT` 和 `IMPORTANT` 等同于 `TIP`，`CAUTION` 等同于 `WARNING`）。有关完整参考，请参阅 [rehype-callouts 文档](https://github.com/lin-stephanie/rehype-callouts)。

### 可折叠的 callout

在类型后面添加 `-` 可以使 callout 默认折叠，添加 `+` 则可以使其默认展开但可折叠：

> [!WARNING]- 继续前请阅读
> 此内容在读者展开之前是隐藏的。适用于可能会打断阅读流程的冗长警告。

> [!TIP]+ 小技巧（默认展开）
> 此内容初始展开，但可以被折叠。适用于你希望在首次加载时可见的可选细节。

### 自定义标题

通过在类型后面添加文本，将默认类型标签替换为你喜欢的任何标题：

> [!NOTE] 你知道吗？
> 类型后面的文本会成为 callout 的标题。如果不写，则自动使用类型名称。

### 语法总结

```md
> [!NOTE]
> 补充信息。

> [!WARNING]- 默认折叠
> 展开前隐藏。

> [!TIP]+ 默认展开但可折叠
> 初始为打开状态。

> [!DANGER] 自定义标题
> 替换默认标题。
```

## 添加目录

默认情况下，文章不包含目录（TOC）。要添加目录，请在希望其出现的位置写上 `目录` 作为 h2 标题（Markdown 中的 `##`）：

```md
---
# frontmatter
---

以下是在 AstroPaper 博客主题中创建新文章的一些建议、提示和技巧。

<!-- [!code ++] -->
## 目录

<!-- 文章其余部分 -->
```

## 标题

关于标题有一点需要注意。AstroPaper 博客文章使用 `title`（来自 frontmatter）作为文章的主标题。因此，文章中的其余标题应使用 `h2` ~ `h6`。

这条规则并非强制，但从视觉效果、可访问性和 SEO 的角度来看，强烈建议遵守。

## 语法高亮

AstroPaper 使用 [Shiki](https://shiki.style/) 作为默认语法高亮器，并通过 [@shikijs/transformers](https://shiki.style/packages/transformers) 来增强围栏代码块。如果你不想使用这些 transformer，可以将其移除：

```bash
pnpm remove @shikijs/transformers
```

```ts file="astro.config.ts"
// ...
// [!code --:5]
import {
  transformerNotationDiff,
  transformerNotationHighlight,
  transformerNotationWordHighlight,
} from "@shikijs/transformers";

export default defineConfig({
  // ...
  markdown: {
    remarkPlugins: [remarkToc, [remarkCollapse, { test: "Table of contents" }]],
    shikiConfig: {
      themes: { light: "min-light", dark: "night-owl" },
      defaultColor: false,
      wrap: false,
      transformers: [
        transformerFileName(),
        // [!code --:3]
        transformerNotationHighlight(),
        transformerNotationWordHighlight(),
        transformerNotationDiff({ matchAlgorithm: "v3" }),
      ],
    },
  },
  // ...
});
```

## 为博客内容存储图片

这里介绍两种存储图片并在 markdown 文件中使用它们的方法。

> [!IMPORTANT]
> 如果你需要在 markdown 中对优化后的图片进行样式设置，应该[使用 MDX](https://docs.astro.build/en/guides/images/#images-in-mdx-files)。

### 存放在 `src/assets/` 目录中（推荐）

你可以将图片存放在 `src/assets/` 目录中。这些图片将通过 [Image Service API](https://docs.astro.build/en/reference/image-service-reference/) 由 Astro 自动优化。

你可以使用相对路径或别名路径（`@/assets/`）来引用这些图片。

示例：假设你想显示路径为 `src/assets/images/example.jpg` 的 `example.jpg`。

```md
![something](@/assets/images/example.jpg)

<!-- 或者 -->

![something](../../assets/images/example.jpg)

<!-- 在 markdown 中使用 img 标签或 Image 组件是无效的 ❌ -->
<img src="@/assets/images/example.jpg" alt="something">
<!-- ^^ 这是错误的做法 -->
```

> [!TIP]
> 技术上来说，你可以将图片存放在 `src` 下的任意目录中。`src/assets` 只是一个推荐做法。

### 存放在 `public/` 目录中

你可以将图片存放在 `public/` 目录中。需要注意的是，存放在 `public/` 中的图片不会被 Astro 处理，这意味着它们不会被优化，你需要自行处理图片优化。

对于这些图片，请使用绝对路径。它们可以通过 [markdown 图片语法](https://www.markdownguide.org/basic-syntax/#images-1) 或 HTML `img` 标签来显示。

示例：假设 `example.jpg` 位于 `public/assets/images/example.jpg`。

```md
![something](/assets/images/example.jpg)

<!-- 或者 -->

<img src="/assets/images/example.jpg" alt="something">
```

## 额外提示

### 图片压缩

> [!WARNING]
> 在博客文章中使用图片（尤其是放在 `public/` 目录中的图片）时，请务必先进行压缩。未优化的图片会严重影响页面性能。

推荐的图片压缩网站：

- [TinyPng](https://tinypng.com/)
- [TinyJPG](https://tinyjpg.com/)

### OG 图片

如果文章没有指定 OG 图片，将使用默认的 OG 图片。虽然不是必须的，但建议在 frontmatter 中指定与文章相关的 OG 图片。推荐的 OG 图片尺寸为 **_1200 X 640_** 像素。

> [!TIP]
> 自 AstroPaper v1.4.0 起，如果未指定 OG 图片，系统将自动生成。详情请查看[相关公告](https://astro-paper.pages.dev/posts/dynamic-og-image-generation-in-astropaper-blog-posts/)。
